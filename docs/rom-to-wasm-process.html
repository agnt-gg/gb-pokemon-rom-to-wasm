00 · Abstract

The short version

This is not a conventional emulator UI, and it is not a source-code port. 00c · Crystal / CGB implementation note

Crystal forced the runtime across the DMG → CGB boundary.

Pokémon Crystal is not just another mapper entry. Its header reports 0x0143 = 0xC0, making it a CGB-only title. Supporting it required a first-class CGB mode rather than pretending the ROM was running on DMG hardware.

FeatureRegister / areaImplemented behavior
CGB boot identityA=0x11, CGB modeCrystal is booted through the CGB path when the header flag is 0xC0.
VRAM bankingFF4F / VBKTwo 8KB VRAM banks; BG/window attributes can select tile bank 0 or 1.
WRAM bankingFF70 / SVBKBank 0 fixed and banks 1-7 switchable for D000-DFFF.
CGB palettesFF68-FF6B64-byte BG and OBJ palette RAM with index/auto-increment behavior and 15-bit RGB decode.
Tile / sprite attributesVRAM bank 1 tile map attrs, OAM attr bitsPalette, tile-bank, and flip attributes are consumed by the renderer.
HDMA pathFF51-FF55General DMA copy plus HBlank block-copy path tied to PPU HBlank transitions.

The current Crystal support is intentionally marked experimental. The implementation is ready for browser testing and title/input smoke, but exact CGB priority rules, HDMA timing, and double-speed cycle accounting remain the next correctness frontier.

02 · Frontier significance

The important claim is not that Pokémon runs in a browser. It is that an AI agent crossed the stack from bits to product.

Many people have written Game Boy emulators. Many people have run Pokémon in browsers. The significance of this build is different: an autonomous agent traversed binary analysis, compiler construction, hardware-runtime reconstruction, instruction/block-level debugging, browser audio/graphics/input, save-system UX, generated visual assets, reusable skill creation, and high-end 3D presentation in one continuous loop.

Not a toy demo

Not merely a wrapper

01 · Critical terminology

Not ASM source → WASM. ROM machine code → WASM.

There is an important distinction. Pokémon Red has a famous disassembly project, but this build did not compile that source. It read the actual cartridge image. The bytes in the ROM are already assembled SM83 instructions. The recompiler's job is to decode those bytes back into instruction objects and lift them into WebAssembly.

The pipeline generalizes from Red to Blue/Yellow/Gold/Silver: Pokemon *.gb/.gbc → SM83 decoder → block graph → WAT module → WASM → JS hardware runtime.
Not this:
  pokered.asm → assembler/compiler → wasm

Actually this:
  Pokemon Red.gb bytes → decode opcodes → lift reachable machine-code blocks → wasm
02 · Pipeline

The build path, step by step

1 · Load

Read the local ROM

The dev server reads the user's local .gb file and wraps it in an MMU that knows the Game Boy address map and MBC3 banking.

2 · Seed

Skip copyrighted boot ROM

The CPU registers are initialized to the known post-boot DMG state, then execution begins at cartridge address 0x0100.

3 · Decode

Turn bytes into instructions

The decoder maps raw opcodes into structured operations: mnemonic, operands, length, cycles, branch targets, and terminator status.

4 · Discover

Walk reachable control flow

Starting from cartridge entry and interrupt vectors, the recompiler recursively discovers static basic blocks.

5 · Lift

Emit WebAssembly Text

Each block becomes a WASM function that mutates register globals, calls imported memory/timing functions, and returns the next PC.

6 · Host

Run in BrowserMachine

V8 executes the generated WASM while JS provides the cartridge bus, graphics, sound, timers, input, saves, and interrupts.

03 · Decoder + block graph

Finding executable structure inside raw bytes

The decoder classifies each instruction and, crucially, identifies terminators: JP, JR, CALL, RET, RST, HALT, and STOP. Static targets are queued as new block starts.

// conceptual instruction object
{
  addr: 0x006f,
  mnemonic: "JR",
  operands: [{ kind: "cond", cc: "NZ" }, { kind: "simm8", value: -6 }],
  length: 2,
  cycles: 8,
  cyclesBranch: 12,
  isTerminator: true,
  staticTargets: [0x006b, 0x0071]
}
⚠️

A subtle decoder bug nearly sank the run: the token C was originally parsed as the carry condition even in non-control-flow instructions. That turned INC C, DEC C, and LD C,n into broken operations. The fix was to treat NZ/Z/NC/C as condition codes only for JP/JR/CALL/RET; otherwise C is register C.

04 · Lifting into WASM

Registers become globals. Memory becomes host calls.

The WASM module stores CPU registers as globals: A/F/B/C/D/E/H/L/SP/PC. It does not directly index ROM or RAM. Every load/store calls imported JS functions rb and wb, so bank switching and IO side effects remain faithful.

SM83

LDH ($0F), A
LDH A, ($FF)
LD B, A
RES 0, A
LDH ($FF), A

Conceptual WASM

(call $wb (i32.const 0xff0f) (global.get $A))
(global.set $A (call $rb (i32.const 0xffff)))
(global.set $B (global.get $A))
;; RES via helper/interp path
(call $wb (i32.const 0xffff) (global.get $A))

Each block returns a next-PC. The host dispatches the next block, or falls back to the interpreter for regions that cannot be statically trusted.

06 · Where it got stuck

The scientific part was mostly falsifying wrong explanations.

The project became playable only after comparing hybrid WASM execution against a pure interpreter and repeatedly isolating the first divergence.

False lead: HRAM execution

The machine crashed at 0xFF80, and HRAM was zeros. At first this looked like a missing HRAM-lift problem. The interpreter revealed the truth: HRAM was a symptom, not the first cause.

Lockstep trap

Comparing interpreter mid-instruction/block state against WASM block-end state produced false divergences. The comparison had to be at equivalent block boundaries.

Decoder C bug

Register C was parsed as condition C for non-branch instructions. This broke copy loops and HRAM/OAM setup. Fix: only parse conditions for control-flow mnemonics.

Correctness escape hatch

Non-fixed-ROM regions are interpreted: switchable bank window and RAM-resident code. This avoids lifting stale bytes from banks/RAM that change at runtime.

// key guard idea
const inFixedRom = pc < 0x4000;
if (!inFixedRom) {
  // execute live bytes through interpreter
  // because banked ROM/RAM/HRAM are not safe static-lift targets
}
07 · Graphics: the invisible-NPC breakthrough

Background worked. Sprites were gone. That meant OAM DMA.

The overworld was visible and interactive, but the player, Professor Oak, NPCs, and many object sprites were invisible. That symptom points directly to OBJ sprites. The sprite renderer was already present, but OAM was empty.

Pokémon writes 0xC3 to FF46 every frame, asking the hardware to copy 0xC300–0xC39F into OAM. The MMU originally treated FF46 as an ordinary IO byte.
if (addr === 0xff46) {
  const src = (value & 0xff) << 8;
  for (let i = 0; i < 0xa0; i++) {
    this.oam[i] = this.read((src + i) & 0xffff);
  }
}

After that, OAM went from 0/160 non-zero bytes to live sprite tables, and the player/NPCs/Poké Balls returned.

08 · Save system + audio

Battery saves are not save-states. Both were needed.

Battery save

In-game SAVE writes to cartridge SRAM. The browser autosaves that SRAM to IndexedDB and can import/export standard .sav files.

Save-state

Snapshots CPU registers, MMU memory, banking state, PPU timing, and timer counters. Slots have overwrite protection.

Later, an auto checkpoint was added: when SRAM persistence happens, the system also writes a protected save-state that does not overwrite manual slots 1/2/3.

Audio was implemented as a pragmatic DMG APU: two square channels, one wave channel, one noise channel, envelopes, sweep, panning, master volume, and a WebAudio worklet ring buffer.

14 · Postscript: the hard timing bug

The hardest failure was not speed. It was correct cycles with wrong state.

The most instructive bug appeared after the game was already playable. The WASM path felt slow, but its total cycle count was not meaningfully behind the interpreter. The machine was not globally underclocked; it was in the wrong state because a timing-sensitive transition driver observed hardware and HRAM state at a different block/interruption cadence.

Cycle totals can be right while interleaving is wrong. In a console recompiler, a generated basic block is not only code; it is also a scheduling decision.

The suspicious loop was the title/transition wait region around 0x1d57 and 0x1d5b:

0x1d57  LDH A, ($BA)
0x1d59  AND A
0x1d5a  RET Z
0x1d5b  LD HL, SP+0
0x1d5e  LDH ($BF), A
0x1d61  LDH ($C0), A
0x1d63  LDH A, ($BB)
0x1d65  AND A
0x1d66  JR Z, +20

The first attempted guard around only 0x1d40–0x1d80 was too narrow. That loop was the symptom, not the writer-side cause. HRAM traces showed the relevant transition flags were written by a broader fixed-ROM driver band, including routines around 0x184b, 0x1876, 0x1d9c, and nearby 0x20xx code. Those routines interact with HRAM transition state such as FFBA, FFBB, FFBF, and FFC0.

⏱️

Reliable fix: route the timing-sensitive fixed-ROM driver band 0x1800–0x2100 through the interpreter oracle while leaving the rest of fixed-ROM execution on generated WASM blocks. This removed the roughly 180-frame title/overworld delay without abandoning the hybrid WASM architecture.

Block boundaries are semantic boundaries

In a normal compiler, merging instructions into a basic block is mostly an optimization. In an emulator/recompiler, it can change when interrupts, VBlank, timers, joypad state, LY, and HRAM flags become visible. The final architecture therefore treats static recompilation as a selective tool, not an ideology: use WASM where behavior is stable, and use the interpreter where hardware interleaving is part of the program's meaning.

Validated native fast path

Native WASM where validated

NOP, DI, EI, XOR, LDH, AND, OR, CP, INC, DEC, ADD, SUB, plus safe LD shapes such as register-to-register, memory-to-register, high-IO loads/stores, and immediate-address stores.

Interpreter where correctness requires it

Banked or unsafe code, RAM/unknown code, unvalidated non-terminators, the dangerous LD reg8 <- imm8 shape, and the timing-sensitive 0x1800–0x2100 title/overworld driver band.

Audio coupling: why speed hacks made music wrong

The audio debugging produced a separate lesson. Pokémon's music engine is not an external media stream; it is ROM code running on the same SM83 CPU as gameplay. Speeding up the CPU also speeds up the code that writes FF10–FF3F APU registers: note triggers, envelopes, sweeps, panning, wave RAM, and master volume. Output stretching after the fact cannot fully repair a register stream that was already produced at the wrong cadence.

14 · Boundaries and next research steps

The honest boundary: impressive integration, not yet a fully general verified recompiler.

The current system is a powerful applied demonstration centered on Pokémon Red and DMG-era Game Boy behavior. A broader research platform would require expanded compatibility, stronger equivalence guarantees, independent reproduction, and a public corpus-driven evaluation process.

Generalization

The system is now proven across the five Pokémon GB/GBC-compatible targets: Red, Blue, Yellow, Gold, and Silver. A general GB/GBC framework still needs systematic support across many more ROMs, mappers, and full CGB hardware variants.

Banked code

The hybrid fallback is pragmatic and correctness-minded. A more advanced translator would model bank context automatically and specialize blocks by mapper state.

Equivalence

The project used interpreter oracles and targeted diagnostics, not a formal proof of instruction/block equivalence. Formal or near-formal semantics remain a natural next step.

Reproducibility

The local source, tests, and artifact exist. A public release should include clean-checkout instructions, ROM-path configuration, test commands, and independent audit notes.

The important nuance is that these boundaries do not diminish the engineering result. The build already includes an interpreter oracle, lockstep debugging, targeted instruction tests, OAM DMA tests, APU tests, save-state tests, banking diagnostics, hybrid static/dynamic execution, and a clear static-vs-hybrid architecture. It is a high-frontier autonomous systems-engineering artifact and a concrete foundation for a broader ROM-to-WASM research program.

11 · Lessons for researchers

What this project taught us

  1. Static recompilation needs dynamic humility. The fixed ROM bank can be lifted; banked ROM and RAM-resident code require live interpretation unless the bank/context is modeled.
  2. Block-level comparison can lie. Compare interpreter and recompiled state at equivalent semantic boundaries, not convenient PCs.
  3. Hardware side effects are the game. FF46 OAM DMA was a tiny missing detail with a huge visible symptom.
  4. Token ambiguity matters. One decoder rule for C broke copy loops and cascaded into phantom runtime failures.
  5. The frontend is not decoration. Save UX, overwrite guards, themes, 3D affordances, and clear panels made the experiment usable enough to test.
  6. The revolution is in the operating model. The notable shift is an agent closing the loop from low-level binary debugging to polished artifact, not any single subsystem in isolation.
The most valuable debugging move was not writing more code. It was building smaller oracles: pure interpreter runs, OAM dumps, HRAM dumps, block-boundary lockstep, and narrowly targeted tests.
